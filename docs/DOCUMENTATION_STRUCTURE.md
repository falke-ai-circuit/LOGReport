# Documentation Structure Guide

**Last Updated**: 2025-10-09  
**Purpose**: Define documentation organization and placement rules  
**Status**: Active

---

## 📁 Directory Structure

```
docs/
├── architecture/        # System architecture (ARCH_*.md)
├── blueprints/          # Implementation plans (BLUEPRINT_*.md)
├── technical/           # Technical guides (TECH_*.md)
├── guides/              # User documentation (GUIDE_*.md)
├── roadmap/             # Project planning (ROADMAP_*.md)
├── analysis/            # 🔄 WORKFLOW: ANALYZE phase outputs
├── implementation/      # 🔄 WORKFLOW: IMPLEMENT phase outputs
├── examples/            # Sample data, test files
└── index.md             # Documentation navigation hub
```

## 📝 Document Types

### Core Documentation (Permanent)

#### 🏗️ Architecture (architecture/)
**Pattern**: `ARCH_[system_name].md`  
**Purpose**: System design, decisions, patterns  
**Examples**: 
- `ARCH_command_system.md` - Command execution infrastructure
- `ARCH_memory_system.md` - 4-layer memory hierarchy
- `ARCH_chatmode_orchestrator.md` - 11-phase workflow system

**Sections**: Overview → Architecture → Components → Decisions → Implementation

#### 📐 Blueprints (blueprints/)
**Pattern**: `BLUEPRINT_[feature_name].md`  
**Purpose**: Implementation plans, feature specifications  
**Examples**:
- `BLUEPRINT_bstool_integration.md` - BsTool integration plan
- `BLUEPRINT_context_menu.md` - Context menu specification

**Sections**: Overview → Requirements → Architecture → Plan → Testing → Resources

#### 🔧 Technical (technical/)
**Pattern**: `TECH_[system_name].md`  
**Purpose**: Technical guides, API documentation, configurations  
**Examples**:
- `TECH_commander_window.md` - Commander window technical reference
- `TECH_codegraph_system.md` - Code graph usage and integration
- `TECH_token_management.md` - Token parsing and validation

**Sections**: Overview → Architecture → API → Configuration → Security → Performance

#### 📚 Guides (guides/)
**Pattern**: `GUIDE_[topic].md`  
**Purpose**: User documentation, tutorials, workflows  
**Examples**:
- `GUIDE_user_documentation.md` - Complete user guide

**Sections**: Overview → Getting Started → Concepts → Procedures → Troubleshooting

#### 🗺️ Roadmap (roadmap/)
**Pattern**: `ROADMAP_[scope].md`  
**Purpose**: Project planning, feature tracking, milestones  
**Examples**:
- `ROADMAP_project_planning.md` - Project roadmap and status

**Sections**: Vision → Phases → Features → Timeline → Metrics

### Workflow Documentation (Active Outputs)

#### 🔬 Analysis (analysis/)
**Purpose**: Outputs from orchestrator ANALYZE phase  
**Generated By**: GitHub Copilot orchestrator (Phase 3: ANALYZE)  
**Lifecycle**: Active during development, archived when obsolete  
**Pattern**: `[topic]_analysis.md`, `[topic]_report.md`

**Content Types**:
- Root cause analysis reports
- Pattern investigation findings
- Dependency mapping
- Technical debt analysis
- Optimization opportunities

**Examples**:
- `kilocode_to_github_copilot_transformation.md` - Migration analysis
- `rule_transformation_report.md` - Rule system analysis
- `unified_chatmode_memory_integration.md` - Memory integration analysis

**Retention**: Keep recent (last 30 days), archive older to `docs/archive/analysis/`

#### 💻 Implementation (implementation/)
**Purpose**: Outputs from orchestrator IMPLEMENT phase  
**Generated By**: GitHub Copilot orchestrator (Phase 5: IMPLEMENT)  
**Lifecycle**: Active during development, archived when obsolete  
**Pattern**: `IMPLEMENTATION_SUMMARY_[feature].md`, `IMPLEMENTATION_REPORT_[feature].md`

**Content Types**:
- Implementation summaries
- Feature confirmation reports
- Refactoring documentation
- Code change summaries
- Integration guides

**Examples**:
- `IMPLEMENTATION_SUMMARY_codegraph.md` - Code graph implementation
- `IMPLEMENTATION_REPORT_hierarchical_commands.md` - Command system implementation
- `CONFIRMATION_all_features_working.md` - Feature validation

**Retention**: Keep recent (last 30 days), archive older to `docs/archive/implementation/`

## 🔄 Document Lifecycle

### 1. Creation Phase
**Workflow Outputs** (`analysis/`, `implementation/`):
- Created automatically by orchestrator during ANALYZE/IMPLEMENT phases
- Follow phase-specific patterns
- Include timestamp, task context, findings

**Core Documentation** (other directories):
- Created following template standards (see `templates/document_standards.md`)
- Require YAML metadata, TOC, cross-references
- Follow type-specific patterns (ARCH_, TECH_, etc.)

### 2. Active Phase
**Workflow Outputs**:
- Referenced during active development
- May inform core documentation updates
- Subject to frequent iteration

**Core Documentation**:
- Maintained in sync with codebase
- Updated during DOCUMENT phase (Phase 9)
- Version tracked in YAML metadata

### 3. Consolidation Phase
**Workflow Outputs**:
- After 30 days or task completion: review for consolidation
- Key insights → integrate into core documentation
- Obsolete content → archive to `docs/archive/[type]/`
- **NEVER delete** - always archive for historical reference

**Core Documentation**:
- Periodic consolidation to prevent fragmentation
- Merge duplicate/overlapping content
- Update cross-references

### 4. Archive Phase
**Location**: `docs/archive/[type]/[YYYY-MM]/`  
**Structure**:
```
docs/archive/
├── analysis/
│   ├── 2025-09/
│   └── 2025-10/
└── implementation/
    ├── 2025-09/
    └── 2025-10/
```

**Archival Rules**:
- Move files older than 30 days (unless actively referenced)
- Maintain directory structure in archive
- Update index.md to remove archived file references
- Keep archive organized by month

## 📋 File Placement Rules

### Source Code Documentation
**Location**: `src/{module}/` alongside code  
**Pattern**: Comments, docstrings within source files  
**Not**: Separate markdown files in `docs/`

### Test Documentation
**Location**: `tests/{module}/` alongside tests  
**Pattern**: Test docstrings, fixture documentation  
**Not**: Separate markdown files in `docs/`

### Configuration Documentation
**Location**: `config/` with inline comments  
**Pattern**: YAML comments, JSON schema  
**Also**: `docs/technical/TECH_[system]_configuration.md` for complex configs

### Workflow Logs
**Location**: `logs/workflow_*.md`  
**Pattern**: `workflow_[feature]_[YYYYMMDD_HHMMSS].md`  
**Not**: `docs/` (logs are operational, not documentation)

### Temporary Analysis Files
**Location**: `misc/temp/`  
**Pattern**: `*_temp.md`, `*_scratch.md`, `*_draft.md`  
**Not**: `docs/` (temp files not finalized documentation)

### Scripts and Tools Documentation
**Location**: `misc/scripts/README.md` or inline script comments  
**Pattern**: Script header comments, usage examples  
**Not**: Separate `docs/` files unless complex system

## 🎯 Quality Standards

### All Documentation Files
- ✅ Proper directory placement (follow rules above)
- ✅ Correct naming pattern for type
- ✅ YAML metadata (9 required fields for core docs)
- ✅ Table of contents with #section links
- ✅ Cross-references to related documents (80%+ for core docs)
- ✅ Clear, concise, actionable content
- ✅ Up-to-date with current implementation

### Workflow Outputs (analysis/, implementation/)
- ✅ Clear task context and timestamp
- ✅ Findings/results section
- ✅ References to related code/files
- ✅ Actionable insights or recommendations
- ⚠️ Metadata optional (not required like core docs)
- ⚠️ Template flexibility (phase-specific format)

### Core Documentation (architecture/, technical/, etc.)
- ✅ 100% template compliance (see `templates/document_standards.md`)
- ✅ Size: 350-2000 lines (compress if larger)
- ✅ Sections: 5-10 major sections with emoji markers
- ✅ Metadata: All 9 fields (title, type, category, version, etc.)
- ✅ Cross-references: 80%+ internal links
- ✅ Navigation: Linked from index.md

## 🔧 Maintenance Guidelines

### Weekly Tasks
1. Review `analysis/` and `implementation/` for completed work
2. Extract key insights into core documentation
3. Update cross-references in affected documents
4. Archive files older than 30 days

### Monthly Tasks
1. Audit all core documentation for accuracy
2. Update version numbers and last_updated dates
3. Consolidate fragmented or duplicate content
4. Clean up archive directories (compress old months)

### Per-Feature Tasks
1. **ANALYZE Phase**: Create analysis report in `docs/analysis/`
2. **IMPLEMENT Phase**: Create implementation summary in `docs/implementation/`
3. **DOCUMENT Phase**: Update relevant core docs in `docs/{type}/`
4. **LOG Phase**: Create workflow log in `logs/` (NOT `docs/`)

### Consolidation Triggers
- 10+ files in same topic area
- Duplicate/overlapping content across files
- Difficulty navigating related information
- User feedback about documentation sprawl

## 🚫 Anti-Patterns (Avoid These)

### Documentation Sprawl
❌ Creating new file for every small change  
✅ Update existing core documentation

❌ Multiple files on same topic  
✅ One comprehensive core document per system

❌ Forgetting to update index.md  
✅ Always link new core docs in index.md

### Directory Misuse
❌ Putting workflow outputs in technical/ or architecture/  
✅ Use analysis/ for ANALYZE phase, implementation/ for IMPLEMENT phase

❌ Mixing active workflow outputs with stable core docs  
✅ Clear separation: core docs vs workflow outputs

❌ Creating new directories without structure update  
✅ Follow defined directory structure

### Naming Confusion
❌ Inconsistent naming (IMPL_, SUMMARY_, Report_, etc.)  
✅ Follow type-specific patterns (ARCH_, TECH_, etc.)

❌ Generic names (notes.md, temp.md, doc.md)  
✅ Descriptive names with proper prefixes

❌ Version numbers in filenames (ARCH_system_v2.md)  
✅ Version in YAML metadata, not filename

### Content Issues
❌ Stale documentation not updated with code  
✅ Update docs in DOCUMENT phase (Phase 9)

❌ Missing cross-references between related docs  
✅ 80%+ internal links required

❌ No YAML metadata in core docs  
✅ All 9 metadata fields required
